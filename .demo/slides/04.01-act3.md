# What AI needs to succeed

AI is only as good as the context you give it.

**Give it:**
- A running app it can navigate
- The codebase
- Consistent `data-testid` attributes — or at least stable roles and labels
- Clear documentation: what the app does, who the users are, what workflows matter
- Small, focused tasks — not "write all our tests"

**Without context, AI guesses. With context, it ships.**

# Tips for using AI to generate tests

- Have good documentation — AI tests what it can read
- Add `data-testid` attributes — every stable selector you add makes AI more effective
- Start small, be specific — one flow at a time
- Let AI do the boring work: selectors, test IDs, boilerplate
- Hand over errors to AI for clarification
- Mono-repo projects are easier for AI to navigate

# Everything you built is a specification language

- `data-testid` attributes are **instructions to AI** — stable contracts it can resolve without guessing
- The seed test is **the interface AI bootstraps from** — fixtures, auth, web server
- `specs/approval-workflow.md` is **your quality standard in plain text** — reviewable, auditable
- Page object fixtures are **the vocabulary AI writes tests in**
- The Healer only works because you defined **what stability looks like**

> You didn't write a test suite.
>
> You wrote the specification language that lets AI test on your behalf.

# The shift in practice

::section::

## Before

- Inspect the DOM, find a selector, write a test
- A UI change breaks it — you fix it manually
- QA is a phase at the end of the sprint
- AI writes code, you test it
- Testing is a cost centre

::right::

## After

- Define `data-testid` contracts — AI finds the selectors
- A UI change breaks a selector — the Healer patches it
- QA is continuous, automated, and self-healing
- AI writes code **and tests it** against your standards
- Testing is a quality multiplier
