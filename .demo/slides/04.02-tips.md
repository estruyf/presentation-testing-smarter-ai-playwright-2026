# Split projects? Use a VS Code Workspace

Frontend and backend in separate repos is common — but AI tools only see what's open.

**The problem:** GitHub Copilot's context is scoped to the workspace. If your test project lives in one repo and the app code in another, Copilot doesn't know about component names, API routes, or data shapes.

**The fix:** create a `.code-workspace` file that includes both repos.

```json
{
  "folders": [
    { "name": "frontend", "path": "../my-frontend" },
    { "name": "backend",  "path": "../my-backend"  },
    { "name": "e2e",      "path": "../my-e2e-tests" }
  ]
}
```

Open the workspace → Copilot sees all three → generated tests reflect real selectors and real API contracts.

# Practical recommendations

Things that make the difference between "it kind of works" and "it actually ships".

- **Create a contract** — define data-testid conventions
- **Start small** — one flow, one persona, one agent at a time
- **Review the plan before generating** — the Planner output is readable; check it
- **Let the Healer fix failures** — don't manually patch locators, that's its job
- **Keep documentation current** — stale docs produce stale tests
- **Delegate the boring parts** — selector generation, boilerplate, error messages
- **Use AI for detailed error analysis** — it can read stack traces and suggest fixes better than a search engine

# Realistic expectations

AI is fast. It is not magic.

| Task | With AI |
|------|---------|
| Test plan for a flow | ~30 min |
| Generated tests from plan | ~1 hour |
| Healing a batch of failures | ~1 hour |

**AI is not perfect — yet**
- Output quality depends heavily on context
- API throttling can slow agent loops
- Platform-specific knowledge (mocked APIs, feature flags) needs your input

The goal: **AI handles the routine, you handle the judgement calls.**
